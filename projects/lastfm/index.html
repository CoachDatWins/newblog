Motivation#

This was a university project for the Architectures for Big Data course. The goal is to collect data, in particular user listenings from Last.fm, store it somewhere, in this case on cloud, and find interesting statistics using Spark, for instance top song per day, week, month, top playlists, and so on.

The code is written in Python and can be found here.

Data collection#

The first part is to collect data, Last.fm offers an API that gives information about user listenings, friends etc. So the process can be summed by:

  • Finding a collection of users to crawl
  • Store the data on cloud, paying attention to store only new things each time

The first point is solved by starting with a random user, moving to his friends, and keep going, until we met a specific number, so the process is a breadth first search on the friends graph.

About the second point, we implemented a CDC (Change Data Capture) architecture, the songs are stored as a log table, i.e without a time field, while user listening are a collection of registries, with a timestamp associated.

The main idea of a CDC is to filter all the incoming data, keeping only the fresh one, avoiding a lot of unnecessary space on cloud. The task is trivial if you have a timestamp associated with every data point, for instance the timestamp associated with a user listening, as you must only keep data after a given instant of time. When a time field is missing, the task is a bit more complex, you store data in a key, value fashion, and keep incoming information if:

  • you never found the key before
  • you have the key but the values changed

This is the way we store songs, basically a key is generated, for the sake of simplicity the hash of the author and the title combined, and when a song field changes, for instance the genre, the new version is stored.

All the data is stored on Google Cloud.

Analysis#

The most interesting part of the project is about the analysis of the collected data, it is done using PySpark with a map reduce approach. I report here only three parts, the rest can be seen on the repo.

Finding playlists#

The task is to find the most relevant playlists in the dataset, this is by far the coolest part of the project. From now on playlist is a sequence of songs listened one after the other.

The idea is to consider a graph, where nodes are the user’s listening sessions, and the edges are weighted according to the Jaccard distance of the sets represented by the nodes.

We then filter the edges that have a really low weight and find connected components. The result are a lot of clusters of similar playlists. The final step is to generalize each component by creating a list of songs that appears a lot in the cluster.

Future works#

A lot could be improved about this project, firstly the data collection could be written in something like C++ of Golang, miles faster than Python.

Secondly, the analysis is all performed on a notebook, it should be scheduled storing data in a database or some sort.

Lastly, for the last step we could have implemented one of the algorithms to find frequent itemsets.
